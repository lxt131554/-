# 100人院内并发压测方案

## 一、测试目标
验证系统在 100 名院内用户同时在线、常规操作场景下的稳定性。

## 二、测试工具
推荐：Apache JMeter 5.6+ 或 k6

## 三、测试环境
- 服务器：与生产环境相同配置的测试机
- 数据库：MySQL 8.x，数据量模拟（500项目/3000阶段/5000填报/1000偏差）
- 网络：院内局域网

## 四、测试场景与接口

### 场景1：登录与工作台（权重30%）
| 接口 | 占比 |
|---|---|
| POST /api/auth/login | 10% |
| GET /api/dashboard | 20% |

### 场景2：项目列表与详情（权重25%）
| 接口 | 占比 |
|---|---|
| GET /api/projects?page=1&size=10 | 15% |
| GET /api/projects/{id} | 10% |

### 场景3：填报与审阅（权重20%）
| 接口 | 占比 |
|---|---|
| GET /api/stages/my-tasks | 5% |
| POST /api/stages/{sid}/reports | 10% |
| GET /api/reports/pending | 5% |

### 场景4：台账与支持（权重15%）
| 接口 | 占比 |
|---|---|
| GET /api/deviations?page=1&size=10 | 10% |
| GET /api/supports?page=1&size=10 | 5% |

### 场景5：领导看板（权重10%）
| 接口 | 占比 |
|---|---|
| GET /api/leader-dashboard | 10% |

## 五、压测指标

| 指标 | 目标值 |
|---|---|
| 平均响应时间 | < 300ms |
| P95响应时间 | < 800ms |
| P99响应时间 | < 1500ms |
| 错误率 | < 1% |
| 吞吐量 | > 50 req/s |
| CPU使用率 | < 70% |
| 内存使用 | 稳定无泄漏 |
| MySQL连接数 | < 30（连接池20） |

## 六、并发梯度

| 阶段 | 并发数 | 持续时间 | 目标 |
|---|---|---|---|
| 预热 | 10 | 2分钟 | 验证基本功能 |
| 低负载 | 30 | 5分钟 | 日常使用场景 |
| 目标负载 | 60 | 10分钟 | 高峰期场景 |
| 压力负载 | 100 | 5分钟 | 极限场景 |

## 七、压测前检查清单

- [ ] 生产配置已应用（HikariCP pool-size=20, Tomcat threads=200）
- [ ] 关键表索引已创建（status/project_id/assignee_id等）
- [ ] 列表接口已分页（page/size，max 100）
- [ ] 数据库连接数 max_connections >= 50
- [ ] 日志级别为 INFO（非 DEBUG）
- [ ] 附件上传限制已配置（max 20MB）
- [ ] 测试数据已准备（500项目/3000阶段/5000填报/1000偏差/1000支持）

## 八、压测后检查

- [ ] 无 HTTP 5xx 错误
- [ ] 无数据库连接超时
- [ ] 无内存溢出
- [ ] 慢查询日志无超过 1s 的 SQL
- [ ] 磁盘空间正常（日志/附件）
- [ ] Session 数量正常

## 九、风险降级预案

如压测不通过，优先级降级方案：
1. 限制工作台/看板刷新频率（前端防抖）
2. 为 Dashboard/LeaderDashboard/Statistics 加 Redis 缓存
3. 异步化 Excel 导入导出
4. 数据库读写分离
5. 增加应用实例（水平扩展）
